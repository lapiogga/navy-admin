**재사용 방법 (확정 -- Option A: sysCode prop 래퍼):**

sys05 페이지를 import한 뒤, 각 페이지를 sysCode prop으로 감싸서 API 경로를 변경한다. 구체적으로:
- sys05 페이지가 내부에서 API base path를 사용하는 경우, sysCode prop을 전달하여 `/api/sys06/` 경로로 호출하도록 한다.
- sys06 MSW 핸들러는 `/sys06/` 경로를 처리하므로, 래퍼에서 sysCode='sys06'을 전달하면 데이터가 격리된다.

```tsx
// 방법: sys05 페이지를 직접 import하고 sysCode prop 전달
import RegulationListPage from '@/pages/sys05-admin-rules/RegulationListPage'
import PrecedentHQPage from '@/pages/sys05-admin-rules/PrecedentHQPage'
import PrecedentUnitPage from '@/pages/sys05-admin-rules/PrecedentUnitPage'
import DirectiveListPage from '@/pages/sys05-admin-rules/DirectiveListPage'

// 각 Route element에서 sysCode="sys06" title="해병대 xxx" props 전달
// 만약 sys05 페이지가 sysCode prop을 아직 받지 않으면,
// 간단 래퍼 컴포넌트를 생성:
// function Sys06RegulationList() { return <RegulationListPage sysCode="sys06" title="현행규정" /> }
```

**주의:** read_first에서 sys05 페이지의 실제 구현을 확인하여, sysCode prop 수용 여부를 판단한다. sys05 페이지가 API 경로를 하드코딩(`/sys05/regulations`)한 경우, 래퍼에서 apiBasePath prop을 추가로 전달하거나, sys05 페이지를 수정하여 sysCode를 동적으로 받도록 한다. sys05 페이지 수정이 필요하면 최소한의 변경(baseUrl을 prop으로 받는 것)만 적용한다.
